# Mayfew: A Methodology For Identifying And Predicting The Hotspot Behavior Of Program Execution Phases

Mayfew is the research pipeline in `hotspot-mitigation-prediction-jake-m`. It was developed to study how software behavior maps to thermal behavior on modern processors, with a particular focus on advanced hotspot prediction and the possibility of targeted mitigation. The project combines architectural, power, and thermal simulation with SimPoint-style workload characterization to identify hotspot-prone execution phases. The pipeline ultimately produces a labeled dataset of code intervals and their hotspot-severity for hotspot analysis and predictive modeling.

The technical goal of the pipeline is to identify which parts of a program are likely to create dangerous thermal behavior and what computational properties explain that behavior. To accomplish this goal, Mayfew contains two feature extraction pipelines used on intervals of 6-20 million instructions. The first is based on the microarchitectural counters exposed by SniperSim, a next-generation x86 simulator. The second is based on recreating the assembly code that ran during the sample interval by preserving the execution order, program counter value, and length of basic blocks generated through QEMU CPU emulation. In my thesis, I used the dataset constructed of these features and labels generated by thermal simulation to do feature importance analyses to better understand the sources of hotspot performance. More practically, this sort of dataset can be used to improve hotspot mitigation by creating predictors that can inform NOP insertion, scheduling/migration, DVFS, and targeted cooling, making those solutions more efficient and effective.

## Supporting Tools

Mayfew sits on top of several existing research tools and codebases. The most important of these is [HotGauge](https://github.com/TuftsCompArchLab/HotGauge), which provides the simulation environment that connects performance, power, and thermal analysis. My work on this repository assumes that simulation stack and builds a more structured interval-analysis workflow around it. This repository also builds on a SimPoint-style interval-selection pipeline used in our lab, including work by Duc Nguyen and Paul Berggren. Their work informs the clustering and representative-interval side of the pipeline; Mayfew extends that work into a larger dataflow that ties interval selection to thermal simulation, code-structure recovery, and machine-learning-ready outputs.

## Pipeline and Repository Structure

The repository is organized into four phases, each of which answers a different technical part of the hotspot-prediction problem.

### Phase 1: Performance and Power Simulation

In Phase 1, a workload and CPU model are fed through the first two steps of HotGauge. These steps are Snipersim, which is the performance model, and McPAT, which is the power model. The output is a series of timestamped files that contain microarchitectural and power statistics of the program's execution on the CPU. These timestamped files are separated into intervals in Phase 2.

### Phase 2: Interval Selection

For the purposes of my thesis, the goal of Phase 2 was to get 2-12 intervals of code execution for each workload that each represented a distinct part of program execution. This meant figuring out some way to identify which intervals were most different from intervals that did not share their phase and which intervals were most alike to intervals within their phase. The SimPoint pipeline developed by my lab-mates was my solution to this. SimPoint creates a feature vector for each interval of code execution that is essentially a histogram containing information on how many of each basic block of the workload executed during that phase. A basic block is just the code in between two branch instructions. With a basic block frequency histogram for each interval, SimPoint performs k-means clustering to determine which intervals are most alike. The intervals at the center of these clusters become the representative intervals that I chose.

Choosing a small number of distinct intervals from a large number of intervals that were simulated in Phase 1 was important because thermal simulation is quite expensive, which limited the size of the end dataset. It was therefore important that the dataset was as diverse as possible. Additionally, this representative interval selection would be useful for mitigation strategies. For example, if one wanted to insert NOPs to eliminate hotspots, they could determine the rate at which they needed to add NOPs to each of the cluster representatives and then apply that rate to the other cluster members. This style of phase-aware mitigation is the ultimate goal of Mayfew.

Another part of Phase 2 that is worth noting is the *run_prefix* script. I built this script because the Snipersim performance simulation is also expensive and the Qemu emulation in the SimPoint pipeline is not. The goal of the script was to use the Qemu emulation to see how much Snipersim simulation that I would have to do to get intervals from the different phases of code execution. The important thing was to make sure that within the amount of instructions I simulated in Snipersim that I got intervals from distinct clusters (if not always the cluster representatives). This script allowed me to sweep across a series of "cluster importance thresholds" to see how many instructions I needed to simulate to get at least one interval from each "important" cluster. For example, if the threshold is .05, the intervals of a cluster need to make up at least 5% of the execution of the program for that cluster to be considered "important". If the threshold is higher, there will be fewer important clusters and it will take fewer simulated instructions to get at least one interval from each important cluster. If it is lower, the opposite happens. The script outputs analysis files that show the tradeoff between the number of instructions simulated in Sniper and the distinctiveness of the available intervals from that limited execution time. 

### Phase 3: Thermal Simulation

Phase 3 takes the timestamped power files from Phase 1 and identifies which of those power files belong to the intervals chosen in Phase 2. For each interval, those power files are looped 200-1000 times and then fed into 3d-ice, the thermal simulator. This looping happens to get an approximation of what the thermal behavior of that individual interval would be if it lasted for 2 billion instructions instead of 2-10 million. The thermal simulator outputs time-stamped heat maps that can be parsed into the hotspot performance metrics MLTD and hotspot-severity.

### Phase 4: Feature Extraction and Dataset Creation

Phase 4 collects the performance statistics for each selected interval from phase 1 and also information about the basic blocks that executed within those intervals, which are generated in phase 2, and creates a feature vector that is paired with the thermal output from phase 3. Some of the features are just the counters taken directly from the performance model, but some are more interesting. The most technically difficult features to extract were the features that attempted to quantify structural dependencies within the code executing in a particular interval. This required recovering the exact assembly instructions of basic blocks and then analyzing the dependencies within those blocks of assembly by creating a directed acyclic graph with registers for nodes and dependencies for edges. This allowed me to get proxies for instruction-level parallelism that are unavailable from the Snipersim architecture statistics.

## Feature Construction

One of the main ideas behind this repository is that no single representation of execution is sufficient by itself. Snipersim is useful because it exposes microarchitectural counters and duty-cycle statistics tied to simulated execution. Those features capture how the processor behaved. QEMU-based basic-block tracing and objdump reconstruction are useful because they expose code structure features like instruction mix, execution ordering, and dependency patterns that Snipersim counters don't capture. The resulting feature set includes architectural throughput measures, memory and branch behavior, functional-unit usage, interval-level entropy measures, simple structure proxies, and hybrid features that compare the static intent of the executed code with the dynamic intensity reported by the simulated machine. The exact list of features and labels can be found in a pdf file in this repo.

## Thesis Context

This repository supports my award-winning honors thesis, **Software Techniques for Advanced Hotspot Prediction and Mitigation**. The thesis argues that software behavior has a measurable effect on hotspot formation and that interval-level software-derived features can be used to characterize, and potentially predict, thermal behavior without running a full thermal simulation for every new case.

The thesis has three connected goals. First, it shows in a controlled setting that software-only changes can alter hotspot behavior. Second, it builds a feature-extraction pipeline that connects program structure and execution behavior to interval-level thermal statsistcs. Third, it uses the resulting dataset to explore whether hotspot behavior can be predicted from those software-generated features. The full thesis text is in this repository.
